# 文档类阶段 Review 清单

适用阶段: `requirements`, `prd`, `ux_design`, `architecture`, `stories`, `database`

## 通用检查（所有文档类阶段）

### 完整性

- [ ] 文档包含所有必要章节（无遗漏的标题/节）
- [ ] 无占位符文本（如 "TODO", "TBD", "[待补充]"）
- [ ] 无空白章节（标题下无内容）
- [ ] 关键数据有具体数值（非 "若干", "一些"）

### 一致性

- [ ] 术语在文档内统一（同一概念不用不同名称）
- [ ] 与前序阶段的产出物不矛盾
- [ ] 引用的外部资源/文件确实存在

### 可操作性

- [ ] 需求/任务描述具体，可直接执行
- [ ] 验收标准明确且可测量
- [ ] 无模糊描述（"尽量", "可能", "大概"）

### 格式规范

- [ ] Markdown 标题层级正确（H1 → H2 → H3 顺序递进）
- [ ] 表格格式完整（有表头、对齐）
- [ ] 代码块标注语言类型
- [ ] 链接有效（无死链）

## 阶段特定检查

### requirements 阶段

- [ ] 包含项目背景/动机
- [ ] 目标用户群体已定义
- [ ] 核心功能需求已列出
- [ ] 非功能需求已考虑（性能、安全、可用性）
- [ ] 约束条件已识别

### prd 阶段

- [ ] 用户故事格式正确（As a... I want... So that...）
- [ ] 每个用户故事有验收标准
- [ ] 优先级已标注（P0/P1/P2）
- [ ] 功能范围与 requirements 一致

### architecture 阶段

- [ ] 技术选型有理由说明
- [ ] 系统架构图清晰（组件 + 关系）
- [ ] API 接口定义完整
- [ ] 数据流向清晰
- [ ] 非功能需求在架构中有体现（扩展性、安全性）
- [ ] 目录结构已定义

### stories 阶段

- [ ] 所有 PRD 用户故事已分配到 Sprint
- [ ] 每个 Story 有故事点估算
- [ ] Sprint 容量未超载（建议 15-20 点/Sprint）
- [ ] 依赖关系正确标注
- [ ] 有明确的完成定义（DoD）

### database 阶段

- [ ] 所有实体/表已定义
- [ ] 字段类型和约束完整
- [ ] 关系（1:1, 1:N, N:N）已标注
- [ ] 索引策略已考虑
- [ ] 与架构文档中的数据模型一致
